# AGENTS.md

Notes for Codex working in this repo.

## What this repo is

This repo is the canonical statement of **Industrial Independence Architecture (IIA)** - an architectural principle, not a product or codebase. The README is the artifact. There is no source code yet.

Author: Daniel "Riverman" Caudle (river@riverman.io). ISA-99 (IEC 62443) committee member.

## Vocabulary that matters

These terms have specific meaning in this document. Don't soften, generalize, or substitute synonyms when editing:

- **ACS** (Automation and Control Systems) vs **IT** - the domain boundary is defined by data criticality, not network topology. Do not conflate. Use **ACS**, not IACS - PERA uses both, IIA standardizes on ACS.
- **SRP** (Safety, Reliability, Performance) - vocabulary for *physical action and physics*. Governs the ACS substrate. Reliability and Performance are properties of the physical system; Safety is what the system protects. Attribution: Infracritical.
- **CIA** (Confidentiality, Integrity, Availability) - vocabulary for *information* (records, historical by nature). Governs IT systems and ACS data once it has crossed the boundary and become information.
- **SAIC** (Safety, Availability, Integrity, Confidentiality) - PERA's extension of CIA, prepending Safety. Useful for IT systems that touch safety-critical *information* (historians, audit stores, regulator-facing dashboards). Does NOT govern the ACS itself. **SRP and SAIC are not variants of one ordering - they describe different substrates.** Never frame as priority-ordering variants. See `feedback_iia_srp_vs_cia_is_category_not_ordering.md`.
- **Zero Trust ↔ Managed Trust** - PERA's framing for the IT/ACS boundary. The box terminates Zero Trust on the IT side and begins Managed Trust on the ACS side. Use this phrase when articulating the boundary's security-model transition.
- **The 4Rs** - Response, Resolution, Reliability, Resilience. PERA's basis for placing applications at the right level. Cite when justifying edge-buffer + central-downsampling.
- **Secure edge gateway** - the canonical architectural term for the unit deployed at the head of every zone. Secure boundary between control system data inside (SRP) and information published outward (CIA). Witness + active poll + decentralized historian + secure outbound publish. Devices inside can be any SL; the security boundary lives at the gateway. The secure outbound publish is the only access into the zone from outside. **"The box" / "the unit"** are colloquial shorthand for the same element - use these in informal/narrative prose; use "secure edge gateway" in formal architectural contexts (specs, definitions, leads). Don't introduce other synonyms. The gateway has two physical realizations: **single-box (SL3, the floor)** and **two-box + diode (SL4, the ideal)**. In the two-box realization, the **inside box** sits on the ACS side (witness, poll, historian) and publishes across a **hardware data diode** to the **outside box / outside subscriber** on the IT side; consumers access the outside subscriber, never the inside.
- **Zone** - the unit of industrial operation. Per IEC 62443.
- **PERA+** (pera.net, Gary Rathwell, Entercon, CC BY-SA 4.0) - primary reference architecture IIA aligns with. Don't drop the "+" on PERA+. Adopted from PERA+: 4Rs, CIAD/CIND diagram conventions, Zero Trust ↔ Managed Trust framing, "secure interfaces, not integration."
- **IEC 62443** - security standard. SL1–SL4, FR1–FR7. IIA targets SL3 floor / SL4 via diode.
- **ISA-95** - data-modeling backbone for the IT↔ACS interface. IIA does not enforce ISA-95 at the edge but feeds ISA-95-modeled consumers without friction.
- **CIAD / CIND** - PERA+ diagram conventions adopted for IIA reference deployments. CIAD = Control and Information Architecture Diagram (conceptual). CIND = Control and Information Network Diagram (preliminary, with SL1–SL4 annotations).
- **CESMII i3X** - preferred standardized query interface at the L2↔L3 boundary. OpenAPI under the hood, time-series + static-state in v1 (transactions punted to v2). Source: CESMII (Clean Energy Smart Manufacturing Innovation Institute). Soft-launch / alpha-with-vendors deployment model. The architecture treats i3X as a load-bearing standard alongside MQTT, OPC UA, and Sparkplug B.
- **MCP (Model Context Protocol)** - Anthropic-developed open protocol for AI-agent access to tools and resources. In IIA, MCP servers run *off-box at broader scope* (broker, corporate); the box exposes i3X over mTLS, an MCP server wraps i3X and exposes data to AI agents. MCP's canonical Streamable HTTP transport conflicts with the box's no-HTTP-at-boundary rule, so MCP server placement is always off-box. This is the *connect-first → model-second → AI-third* pattern made concrete (i3X = connect, knowledge graph / ISA-95 = model, MCP + AI agents = third tier).
- **Zenoh** - Eclipse open-source pub/sub + queryable + federated-router protocol. Distinctive in that it can fill multiple IIA roles on one protocol: edge publisher (pub/sub), structured query API (queryables), in-flight bus, and inter-box mesh substrate (cross-scope routing across the fractal). No rebirth-storm or QoS-1/2-degraded-integrity pathology. Apache 2.0 OSS.
- **Inter-box mesh substrate** - distinct role from in-box bus and edge publisher. Enables a box at one scope to query / subscribe to a box at another scope, across the fractal. Optional - explicit upstream-pull patterns work without it. Zenoh federated routers fit this role natively; cascading MQTT brokers and custom mTLS overlays are alternatives.
- **Distributed**, not **federated** - IIA is a distributed mesh of identical units under one operator, not a federation of independent parties exchanging data by treaty. Never describe it as "federated."
- **Sovereignty ≠ isolation** - units are self-contained, but historians on adjacent boxes may know about each other. The constraint is "no dependency," not "no awareness." Don't write language that implies units are mutually invisible.
- **Centralization is not a separate model** - a centralized collector (corporate, regional, cloud) is just another box at broader scope, not a distinct architecture. The fractal does not collapse at the top. Don't write language that treats "centralized" as architecturally different from "edge."
- **Roles, not products (in the spec).** The architecture spec (`docs/internal-architecture.md` normative sections, README) names roles. Specific software does not belong in the architecture documents. See `feedback_iia_roles_not_products.md` and `project_iia_reference_implementations.md`.
- **Data contract** - explicit description of every communication on the box and across its boundaries. Universal: inter-container, inter-zone, cross-side, external, and device-level. The deployment's full set is the **contract catalog**, versioned and discoverable via the structured query API. Communication without a contract is **prevented** (kernel firewall + workload identity + admission policy cooperate; no contract → no SVID → no mTLS → no traffic) or **flagged** (where prevention is impossible - passive ACS observation, broadcast, legacy protocols; emitted as `ot.contract.violation`). Contractlessness at any layer is a deployment defect. **Boundary data contracts** (a specialization, at every connection exiting the ACS) are *bilateral*: ACS commits to producing per spec; upstream commits to connectivity, auth, ack, response, capacity, incident response. Includes a **RACI matrix** for every failure mode. **Adherence telemetry** under `ot.contract.*` records connectivity, delivery, auth, schema, quota, reconciliation, audit verification, SLA breaches, and violations - receipts of who held up which side. Architecture specifies the shape; operator publishes the values. See `docs/internal-architecture.md` *Data Contracts* section. Synonyms: *data contract*, *bilateral data contract* (boundary case), *SLA*, *consumer contract* (when single-direction is sufficient).
- **Orchestrate / choreograph** - verbs for how flows behave. Transactional flows (MES/MOM ops) are *orchestrated*: synchronous, ack'd. Telemetry flows are *choreographed*: fire-and-forget. Mixing in one pipeline is a recurring failure mode. The box's edge publisher handles telemetry; transactions go through the structured query API or a dedicated transactional channel.
- **Attestation** - independent observation that validates whether prevention is actually working. Three layers: **SDN-policy attestation** (network plane policy compiled from the contract catalog enforces; IDS cross-checks observed traffic against the catalog), **IDS as policy-leak observer** (`ot.attestation.network` flags traffic that no contract authorizes, expected traffic that doesn't appear, mTLS sessions out of tolerance, policy drift), **IO master** (`ot.attestation.io` - independent observation of the physical IO substrate including analog, fieldbus, industrial Ethernet; cross-checks the box's primary capture; discrepancies indicate substrate-level deviation). All attestation findings emit under `ot.attestation.*` and route to the audit chain at the same severity as security events. The architecture's standing answer to *"how do you know prevention is actually working?"* See `docs/internal-architecture.md` *Attestation* section.
- **IO master** - a role: independent IO substrate observer running in its own zone (`ot.acs.io_master`) with its own SPIFFE identity and attestation key. Cross-checks readings against `ot.acs.collect` / `ot.acs.witness`. Independence of observation path is the load-bearing property - sharing a NIC / switch port / software stack with the primary defeats the attestation.
- **SDN policy plane** - an *optional* prevention layer (recommended where available): programmable network policy compiled from the contract catalog, enforced on the fabric. Where deployed, conduits exist on the wire only where the catalog authorizes. Cilium/eBPF on appliance scope; OVN-class on multi-host. **Non-SDN deployments are fine** - they rely on the host-layer prevention triad (kernel firewall + SPIFFE + admission) + mTLS + **the IDS attestation observer being able to alert** as the network-plane-only observability layer. Architecture's hard requirement is that *some* network-plane attestation can alert; SDN strengthens prevention but doesn't replace attestation.
- **Configuration as signed artifact** - the box accepts configuration only as a signed, constrained-grammar text artifact consumed by a parser, never as a live API. Same pattern as image admission and OS updates. The management UI is a text generator with no privileged access. The parser is the trust boundary. The applier (gated internal call set) executes when a verified+validated artifact is staged, then exits. **The box has no live mutation channels** - image, OS, and configuration are all artifact-in flows. Forensic investigation can reconstruct every state transition: artifact hash → parser validation → applier calls → before/after state, signed by operator, hash-chained. Pattern is GitOps for an air-gapped industrial appliance. Configuration attestation observer cross-checks running state vs staged artifact and emits `ot.attestation.config` divergence events. See `docs/internal-architecture.md` *Configuration* section.
- **Parser as trust boundary** - for configuration what container-runtime admission is for workloads, and what cosign verification is for images. Constrained, non-Turing-complete grammar required (small parser is auditable; Turing-complete grammar loses the security property). Operator-chosen format (CUE, KCL, Dhall, JSON-Schema-validated JSON, etc.).
- **Update orchestration is fractal.** Each box at broader scope manages its children's updates: publishes intended-state manifest (OS image hash, config hash, rule-set hashes, approval artifacts), receives reported state (running hashes, attestation, audit). No global update server - fleet's source of truth lives at whatever scope the operator manages from. Children pull manifests and approvals on a cadence; reconcile to declared state; report back. Pull-only, mTLS, no inbound listener, no outbound HTTP. **Approval artifacts** authorize specific boxes to apply specific staged updates within specific time windows; supports per-box, per-fleet, two-person integrity (FR2 SR 2.4), maintenance-window-conditional. Rollout strategy (canary / percentage / per-zone / per-plant) is operator policy expressed via *which approval artifacts get issued to which boxes when*; architecture provides substrate, operator provides policy. Disconnected boxes converge on intended state when reconnected, at operator pace via still-valid approvals. Image attestation (`ot.attestation.image`) closes the post-update reconciliation loop. Emergency rollback via revert approval artifact (remote) or TPM-bound serial console (local last-resort). See `docs/internal-architecture.md` *Update Orchestration* section.

## Architectural decisions on file

Locked in across the 2026-05-02 → 2026-05-04 design conversations. Don't relitigate without explicit invitation. These are spec-level decisions - architecture invariants. Operator-side engineering decisions (which software fills which role) are not tracked here and are not part of the architecture.

### Architecture-spec invariants (in normative documents)

- **SL3 floor / SL4 via two-box + diode.** Same architecture across both; only physical topology changes. SL1/SL2 are SL3 with controls relaxed by deployment policy, not separate products.
- **SL3 does NOT enforce unidirectional flow.** SL3 is segmentation + authentication + audit + push-only outbound on operator-selected edge profile. Bidirectional channels exist at SL3 (the structured query API listener, the outbound tunnel reverse-dial); they are firewalled, identified, recorded. Unidirectional flow is the SL4 promise via hardware diode.
- **The Two-Box Method is credited.** Rinaldi & Workman, *The Everyman's Guide to EtherNet/IP Network Design*, RTA 2022, ISBN 9798839986152. IIA generalizes their pattern in software (SL3) and deploys natively into it (SL4). README has a section on this.
- **No HTTP at the box's external boundary, in either direction, at runtime.** No HTTP listener on ACS or IT NIC. No outbound HTTP/HTTPS from any component. Updates and deltas come via signed bundles or mTLS-tunneled deltas. The optional local management UI on the management NIC is the only HTTPS listener anywhere - local-network only.
- **Minimum runtime surface.** Every listener, daemon, outbound connection exists only because designed and engineered for a specific operational purpose. No default-on services. The OS image is built so disabled services cannot be turned on at runtime.
- **ACS NIC is passive only.** No IP stack TX, no listeners ever.
- **Three-side internal partitioning + management interface.** Inbound (ACS-facing) / Internal DMZ / Outbound (IT-facing) / Management. Inbound and outbound zones cannot conduit directly to each other; all cross-side traffic transits the DMZ.
- **Local data lake is source of truth on the box.** All inbound capture, classification output, audit, events, time-series land there. Outbound publishers siphon from the lake. The in-flight bus in the DMZ is transient and durable nothing.
- **Edge profile is operator-selectable per deployment.** No named default. MQTT+Sparkplug B is valid only at controller↔area-broker (PERA L1/L2). Above L2, prefer OPC UA pub/sub, structured query API on mTLS, or batch-write to BI lake.
- **Structured query API is mTLS-authenticated, non-HTTP.** gRPC, NATS req/reply, MQTT req/reply, OPC UA, MCP-over-mTLS - all valid candidates per deployment.
- **No schema enforcement at the edge.** Box publishes typed-but-flexible records; modeling lives at broader scope.
- **Hardware reference: 8 GB / 4 cores / 1 TB SSD / TPM 2.0 for SL3.** 3× 1GbE ideal (ACS / IT / Management); 2× 1GbE + VLAN acceptable on commodity hardware. The 4 GB / 2 core figure in the README is the architectural floor / SL2 reference. Don't change README numbers without asking.
- **PERA+ alignment.** Adopted: 4Rs, CIAD/CIND diagram conventions, Zero Trust ↔ Managed Trust framing, "secure interfaces, not integration." Credit: Gary Rathwell, Entercon, pera.net.
- **Architecture is hardware-agnostic and scale-invariant.** "The box" is a logical role-aggregate. Realizable as appliance, multi-host, virtualized, k8s cluster, or hyperscale - invariants travel across all realizations. The commodity-hardware appliance SKU is positioning for accessibility, not a constraint of the principle. "No k3s" applies to the appliance SKU only; multi-host realizations are free to use k3s, full k8s, Nomad, or any orchestrator that fits.
- **Field-consensus framings absorbed (Breakfast Club).** Sparkplug B at L1↔area-broker only. Edge profile operator-selectable; no named default. Schema observed, not enforced, at the edge. Knowledge-graph audit + microservices over broker payloads at broader scope. Data contracts are the L2↔L3 integration unit. Orchestrate transactions, choreograph telemetry - never mix in one pipeline. ISA-95 is the L3 modeling backbone. Don't integrate, query (I3X / MT Connect / MCP / Zenoh). BI is its own pipeline (Iceberg / Delta / DuckLake on S3). Wireless is acceptable for IO, not control. Top-down + bottom-up governance both required (Walker red/blue, David's medallion / "thou shalt vs thou may"). Connect first, model second (ISA-95), AI third. Never accept an integrator-built closed MES. See `feedback_iia_field_consensus_breakfast_club.md` for the full set with attribution.
- **The internal architecture lives in `docs/internal-architecture.md`.** Treat that file as the canonical implementation spec for what runs inside the box. Names roles, not products.
